The LimeLite

The LimeLite is Better Way Lighting's in-house developed tungsten and daylight panel light that's 65% lighter than the industry standard panels. The LimeLite utilizes a revolutionary lime diode and advanced algorithms to eliminate the magenta shift found on competing fixtures. The LimeLite is also designed to fit existing grids and light control accessories designed for the SkyPanel or Vortex8.

I worked on the design, layout, and behaviour of the LimeLite's UI, as well as connecting the UI to the data model and device backend. I also added new data model entries and backend functionality as needed to implement new features or modify existing features and UI behaviour.

UI Design and Features

Initial UI Design and Layout

I started designing the UI by first analyzing features from competing panel lights and narrowing down options we wanted to have based on the hardware capabilities. Using the list of features, I designed UI mockups with our in-house card-based UI library, which we also use on devices with smaller versions of the LimeLite's screen.

Feature and Layout Selection

After discussing the UI mockups with the team, we finalized the software and UI features we wanted. Based on the new feature list, created flow charts for potential UI layouts using FigJam, leading to the selection of a single menu layout that updates to show control mode-specific settings based on the selected control mode.

Using the UI mockups, I test-coded test versions of the UI cards with and without the menu title to roughly determine the layout of the cards and the font sizes. Based on the feedback from the team, we determined the layout of the card elements and chose font sizes that allowed us to fit at least 3 lines of text on the display. Based on the test versions of the UI cards, I created a mockup list of the basic card types and font size options. The card type mockup list was then sent to the graphic designer to finalize the design of the UI cards.

Building the UI

Initial UI Coding

Using the selected UI layout and the finalized UI card designs, I began modifying the UI library on a separate branch to fit the larger screen size and coding the LimeLite UI in a separate code repository while the backend and data model was being built. I also adapted existing custom cards we needed from the other devices using the same UI library and created new custom cards as needed. While coding the LimeLite UI, I also added dummy callback functions to the cards that require them in preparation for connecting the UI with the data model and backend.

Linking the UI

Once the initial UI, data model, and backend were created, I began linking the UI to the data model and backend. During the linking process, I reorganized the UI and removed unnecessary cards. Additionally, I added new UI cards and display elements for features in the backend that were'nt in the initial UI. I also assisted in adding any data model entries and backend functionality for the UI cards that had not been implemented yet.

Adding New Features

After linking the initial UI to the initial data model and backend, I helped implement new major features and adjust the behaviour of existing features and the UI. After the major features were added, I then assisted in adding minor features for release, with ongoing modifications to the UI, data model, and backend to accommodate desired functionalities and device behaviors.

Some of the features I added or assisted in adding include the adjustable lock timeout, warning and status modals, navigating to the status screen on encoder scroll and UI lock, device temperature display, and the ability to set the flash button output to the current light output. I also helped add development-only behaviour and UI cards to make it easier to test the device.

Colour Temperature Lookup Tables

One major feature I helped develop along side the other co-op student there at the time was the lookup tables for the colour temperature adjustments. Since colour temperature is logarithmic in terms of the actual value and the perceived colour change, we decided on making the colour temperature adjustments based on the perceived colour change for smoother adjustments.

Using the python colour-science library, we began by plotting colour temp values on the Planckian locus of the colour temp x/y graph and the visibly similar colour temp ranges for each value. Seeing that the ranges were rotated ellipses, I took the rotated ellipse equation and used it to determine the exact visibly similar colour temp range for a given colour temp, which we then used to modify the colour temp plot values so that the ranges don't overlap.

Once the colour temp values were determined, we then generated the lookup table, which I then used to modify the UI and light output so the colour temp adjustments and the displayed UI value match the lookup table. We then re-calculated the lookup table for high brightness mode once it was added, and I added the functions to convert the light output and displayed UI value when switching between normal and high brightness mode.
